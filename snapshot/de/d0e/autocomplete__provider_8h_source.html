126 // (A search keyword is a keyword with a replacement string; a bookmark keyword
127 // is a keyword with no replacement string, that is, a shortcut for a URL.)
128 //
129 // There are two possible providers for search suggestions. If the user has
130 // typed a keyword, then the primary provider is the keyword provider and the
131 // secondary provider is the default provider. If the user has not typed a
132 // keyword, then the primary provider corresponds to the default provider.
133 //
134 // Search providers may supply relevance values along with their results to be
135 // used in place of client-side calculated values.
136 //
137 // The value column gives the ranking returned from the various providers.
138 // ++: a series of matches with relevance from n up to (n + max_matches).
139 // --: a series of matches with relevance from n down to (n - max_matches).
140 // *: relevance score falls off over time (discounted 50 points @ 15 minutes,
141 // 450 points @ two weeks)
142 // **: relevance score falls off over two days (discounted 99 points after two
143 // days).
144 // +-: A base score that the provider will adjust upward or downward based on
145 // provider-specific metrics.
146 //
147 // A single result provider for the autocomplete system. Given user input, the
148 // provider decides what (if any) matches to return, their relevance, and their
149 // classifications.

181  // Called to start an autocomplete query. The provider is responsible for
182  // tracking its matches for this query and whether it is done processing the
183  // query. When new matches are available or the provider finishes, it
184  // calls the controller's OnProviderUpdate() method. The controller can then
185  // get the new matches using the provider's accessors.
186  // Exception: Matches available immediately after starting the query (that
187  // is, synchronously) do not cause any notifications to be sent. The
188  // controller is expected to check for these without prompting (since
189  // otherwise, starting each provider running would result in a flurry of
190  // notifications).
191  //
192  // Once Stop() has been called, usually no more notifications should be sent.
193  // (See comments on Stop() below.)
194  //
195  // |minimal_changes| is an optimization that lets the provider do less work
196  // when the |input|'s text hasn't changed. See the body of
197  // OmniboxPopupModel::StartAutocomplete().

200  // Advises the provider to stop processing. This may be called even if the
201  // provider is already done. If the provider caches any results, it should
202  // clear the cache based on the value of |clear_cached_results|. Normally,
203  // once this is called, the provider should not send more notifications to
204  // the controller.
205  //
206  // If |user_inactivity_timer| is true, Stop() is being called because it's

265  // Finds the matches for |find_text| in |text|, classifies those matches,
266  // merges those classifications with |original_class|, and returns the merged
267  // classifications.
268  // If |text_is_search_query| is false, matches are classified as MATCH, and
269  // non-matches are classified as NONE. Otherwise, if |text_is_search_query| is
270  // true, matches are classified as NONE, and non-matches are classified as
271  // MATCH. This is done to mimic the behavior of SearchProvider which decorates
272  // matches according to the approach used by Google Suggest.
273  // |find_text| and |text| will be lowercased.
274  //
275  // For example, given
276  // |find_text| is "sp new",
277  // |text| is "Sports and News at sports.somesite.com - visit us!",
278  // |text_is_search_query| is false, and
279  // |original_class| is {{0, NONE}, {19, URL}, {38, NONE}} (marking
280  // "sports.somesite.com" as a URL),
281  // Then this will return
282  // {{0, MATCH}, {2, NONE}, {11, MATCH}, {14, NONE}, {19, URL|MATCH},
283  // {21, URL}, {38, NONE}}; i.e.,
284  // "Sports and News at sports.somesite.com - visit us!"
285  // ^ ^ ^ ^ ^ ^ ^
286  // 0 2 11 14 19 21 38
287  // M N M N U|M U N
288  //
289  // For example, given
290  // |find_text| is "canal",
291  // |text| is "panama canal",
292  // |text_is_search_query| is true, and
293  // |original_class| is {{0, NONE}},
294  // Then this will return
295  // {{0,MATCH}, {7, NONE}}; i.e.,
296  // "panama canal"
297  // ^ ^
298  // 0 M 7 N

300  const std::u16string& find_text,
301  const std::u16string& text,
302  const bool text_is_search_query,
303  const ACMatchClassifications& original_class = ACMatchClassifications());

335  // Fixes up user URL input to make it more possible to match against. Among
336  // many other things, this takes care of the following:
